# Swell 桌面视频下载器设计说明

日期：2026-06-20  
状态：待评审  
范围：新建独立仓库，开发一个面向个人使用、以成功率优先的桌面 GUI 视频下载器

## 一、摘要

本项目不再沿用当前 Chrome MV3 扩展仓库作为主产品形态，而是新建一个独立的桌面应用。产品形态为 GUI 优先的视频下载器，用户输入视频页面 URL 后，即可解析并下载视频。第一阶段优先支持：

- `x.com`
- `pornhub.com`

本产品的核心目标不是“尽可能低的安装门槛”，而是“自己日常使用时尽可能稳定可用”。

第一版的理想体验应该是：

- 粘贴一个视频页面地址
- 一键下载系统推荐的最佳版本
- 需要时可以展开查看更多清晰度/格式
- 可以看到下载进度，并在失败时重试

本项目不自己重写整套站点解析与媒体合流能力，而是基于成熟工具链实现：

- 使用 `yt-dlp` 负责站点解析、格式枚举与下载
- 使用 `ffmpeg` 负责音视频合流、转封装和后处理
- 使用桌面应用外壳负责本地文件系统访问、浏览器 Cookie 读取、任务状态管理

## 二、产品决策

当前仓库不继续作为主产品线演进。

原因如下：

- 当前项目是围绕 MV3 扩展模型组织的，核心包括 service worker、content script、`chrome.*` API、offscreen document 等。
- 新目标是桌面 GUI 应用，核心能力是本地命令执行、原生文件访问、浏览器 Cookie 读取、持久化下载任务管理。
- 如果把两者混在一个代码库里，运行时边界会变得模糊，构建和维护成本也会持续升高。

建议策略：

- 当前仓库保留为浏览器扩展原型和参考实现
- 新建一个独立仓库，作为桌面下载器正式产品线
- 等桌面端稳定后，如有需要，再追加“扩展把链接发给桌面端”的联动能力

## 三、目标用户与目标

主要用户：

- 项目作者本人，作为日常自用工具

核心目标：

- 成功率优先于安装门槛
- 对已支持站点提供尽可能顺手的使用流程
- 尽量减少重复操作步骤
- 在失败时给出清晰错误和恢复路径

次级目标：

- 以 Windows 体验为优先
- 提供基础 macOS 支持
- 为未来浏览器扩展联动预留空间

第一版非目标：

- 不是一个“任意网站全能下载器”
- 不以内嵌抓包或浏览器嗅探作为主要下载路径
- 不做账号、频道、播放列表的批量抓取
- 不做云同步、远程任务或多人协作能力

## 四、第一版支持范围

明确优先支持：

- `x.com`
- `pornhub.com`

策略要求：

- 第一版只针对这两个站点优化成功率和使用体验
- 在这两个站点稳定之前，不主动扩展到“任意视频网页都能下”

## 五、总体架构

整体采用基于 Tauri 的桌面 GUI 应用形态，前端使用 React。

分层如下：

1. 前端界面层  
   负责 URL 输入、解析结果展示、高级格式选择、下载队列、设置页、Cookie 引导。

2. 桌面命令层  
   负责本地命令调用、文件系统访问、浏览器 Cookie 读取、进度事件回传、依赖检查。

3. 下载引擎层  
   使用 `yt-dlp` 作为解析与下载主引擎。

4. 媒体处理层  
   使用 `ffmpeg` 负责合流、转封装及必要后处理。

5. 平台适配层  
   负责处理 Windows 和 macOS 在浏览器配置目录、Cookie 读取、路径规则上的差异。

## 六、技术选型

### 1. 桌面外壳

使用 Tauri。

原因：

- 比浏览器扩展更适合本地命令执行与文件系统访问
- 相比 Electron 更轻量
- Windows 支持友好
- 原生命令与事件桥接模型适合下载任务型产品

### 2. 前端

使用：

- React
- TypeScript
- Vite
- React Spectrum 2（`@react-spectrum/s2`）

原因：

- 组件体系完整，适合桌面工具类产品
- 表单、列表、进度、弹窗、可访问性能力成熟
- 很适合“输入 + 结果列表 + 下载队列 + 设置页”这类工具型交互

### 3. 下载核心

使用 `yt-dlp`，不自行实现站点提取器。

原因：

- 对 `x.com` 与 `pornhub.com` 已有成熟支持经验
- 在认证、格式枚举、失败回退方面远比自研更稳
- 能显著降低第一版实现风险

### 4. 媒体处理

使用 `ffmpeg`。

原因：

- 对音视频分离流的合并与转封装能力可靠
- 与 `yt-dlp` 协作模式成熟
- 是输出最终单文件结果的必要基础能力

## 七、交互与界面设计

第一版应保持“紧凑、实用、顺手”，不要演变成媒体管理平台。

### 1. 主页面

主页面是第一核心入口，应包含：

- 一个显眼的 URL 输入框
- `下载最佳` 按钮
- `查看更多格式` 按钮
- 最近一次解析结果的简要信息

### 2. 解析结果区

当 URL 解析完成后，展示：

- 标题
- 封面（可获取时展示）
- 时长（可获取时展示）
- 作者 / 账号名（可获取时展示）
- 系统推荐格式
- 可展开的高级格式列表

高级格式列表每一项应展示：

- 分辨率
- 封装格式
- 是否包含音频
- 预估大小（可获取时展示）
- 补充说明，例如编码信息或备用说明

### 3. 下载队列

下载队列应始终可见，或能非常方便地切换到。

每个任务展示：

- 视频标题
- 当前状态
- 当前进度
- 下载速度
- 预计剩余时间（可获取时展示）
- 操作按钮：重试、取消、打开目录

### 4. 设置页

第一版设置项保持克制，只保留最有价值的项：

- 默认下载目录
- 默认下载策略
- 文件命名规则
- 下载完成后是否自动打开目录
- Cookie 来源配置
- `yt-dlp` / `ffmpeg` 依赖状态

## 八、核心用户流程

### 流程 A：一键下载最佳版本

1. 用户粘贴支持站点的视频页面 URL
2. 应用校验 URL 并开始解析
3. 应用获取元数据和格式列表
4. 应用选出推荐的最佳格式
5. 用户点击 `下载最佳`
6. 任务进入下载队列
7. 下载完成
8. 如有必要，自动执行合流或转封装
9. 最终文件保存到目标目录

### 流程 B：手动选择高级格式

1. 用户粘贴支持站点 URL
2. 应用解析元数据和可用格式
3. 用户展开高级格式列表
4. 用户手动选择某个格式
5. 任务进入队列并开始下载
6. 如需要，自动执行后处理

### 流程 C：Cookie 失败回退

1. 应用先尝试使用浏览器 Cookie 自动访问目标站点
2. 如果失败，应用提示可能原因
3. 引导用户导入 Cookie 文件
4. 应用校验导入结果
5. 重新执行解析或下载

## 九、Cookie 策略

这是决定成功率的关键模块之一。

### 1. 主路径：自动读取本机浏览器 Cookie

优先级如下：

- Windows：优先支持 Chrome 和 Edge
- macOS：优先支持 Chrome，后续视情况补 Edge

行为要求：

- 用户可以在设置页里选择默认浏览器来源
- 对支持站点进行解析和下载时，默认自动带上配置好的 Cookie 来源

### 2. 备用路径：手动导入 Cookie

要求：

- 支持导入与工具链兼容的 Cookie 文件
- UI 中提供明确的获取说明
- 导入后必须先校验，再提示用户“可用于当前站点”

### 3. 设计原因

自动读取浏览器 Cookie 可以极大降低日常使用门槛；手动导入则能覆盖路径差异、权限限制、浏览器加密策略、环境异常等场景，避免整条链路被单点失败卡死。

## 十、下载任务状态模型

每个任务都应有清晰的状态机：

- `pending`：等待中
- `resolving`：解析中
- `ready`：已准备
- `downloading`：下载中
- `postprocessing`：后处理中
- `completed`：已完成
- `failed`：失败
- `canceled`：已取消

规则要求：

- 解析失败必须以“解析错误”形式结束
- 下载失败必须允许重试
- 后处理失败时，尽量保留中间产物
- 取消状态必须与失败状态清晰区分

## 十一、错误处理要求

应用不能静默失败，必须尽可能透明。

必须具备以下行为：

- 清楚展示失败发生在解析、下载还是后处理阶段
- 为每个失败任务保留可读错误摘要
- 允许复制完整日志，方便排查
- 自动 Cookie 读取失败时，立即引导到手动导入路径
- 推荐格式下载失败时，提示用户尝试高级格式中的其他版本
- 合流失败时，在可行情况下保留原始文件并说明下一步建议

## 十二、React Spectrum 组件规划

界面统一使用 React Spectrum 2。

建议组件映射：

- URL 输入：`TextField` 或 `SearchField`
- 主操作按钮：`Button`
- 次级操作按钮：`ActionButton`
- 结果摘要卡片：`Card`
- 高级格式列表：`ListView` 或 `TableView`
- 下载队列：`TableView`
- 状态指示：`StatusLight`
- 错误与警告：`InlineAlert`
- 进度展示：`ProgressBar`
- 设置项选择：`Picker`、`Switch`
- 弹窗流程：`Dialog`
- 主分区切换：`Tabs`

## 十三、新仓库目录结构建议

推荐的新仓库结构如下：

```text
src/
  app/
  features/
    resolve/
    downloads/
    settings/
    cookies/
  components/
  lib/
src-tauri/
  src/
    commands/
    downloader/
    platform/
    cookies/
    events/
docs/
bin/
```

含义如下：

- `src/app`：应用壳、Provider、路由、主题、顶层布局
- `src/features/resolve`：URL 输入、结果卡片、推荐格式与高级格式界面
- `src/features/downloads`：下载队列、状态渲染、重试与取消动作
- `src/features/settings`：下载目录、默认行为、依赖状态与 Cookie 设置
- `src/features/cookies`：浏览器来源选择、手动导入流程、说明引导
- `src/lib`：前端共享类型、命令封装、事件订阅工具
- `src-tauri/src/commands`：Tauri 命令入口
- `src-tauri/src/downloader`：解析、入队、执行、重试、取消、后处理
- `src-tauri/src/platform`：系统差异适配逻辑
- `src-tauri/src/cookies`：Cookie 发现、导入与校验
- `src-tauri/src/events`：前端事件推送

## 十四、后端命令面设计

前端与桌面层之间保持一组小而稳定的命令接口：

- `resolve_media(url)`
- `start_download(payload)`
- `cancel_download(task_id)`
- `retry_download(task_id)`
- `list_cookie_sources()`
- `select_cookie_source(source_id)`
- `import_cookie_file(path)`
- `check_dependencies()`
- `pick_download_directory()`

下载过程中的状态变化通过事件通道推送：

- `download://progress`
- `download://status`
- `download://error`

## 十五、第一版范围

第一版包含：

- 解析 `x.com` 与 `pornhub.com` 页面 URL
- 一键下载推荐的最佳版本
- 展示并支持选择高级格式
- 下载队列与进度展示
- 重试与取消
- 自动读取浏览器 Cookie
- 手动导入 Cookie 作为兜底
- 必要时自动执行合流 / 转封装

第一版不包含：

- 账号、频道、播放列表级别的批量抓取
- 内置浏览器
- 通用网页嗅探模式
- 复杂的历史数据库
- 多设备同步
- “支持全站通用下载”的产品承诺

## 十六、交付策略

第一阶段目标是尽快做出一个“单链接高可用”的可用产品。

建议开发顺序：

1. 搭建 `Tauri + React + TypeScript + React Spectrum`
2. 搭建基础 GUI：输入区、结果区、队列区、设置入口
3. 实现 `resolve_media`，跑通目标站点解析
4. 实现 `下载最佳`
5. 实现下载进度与失败展示
6. 实现自动 Cookie 读取
7. 实现手动 Cookie 导入兜底

当作者可以不借助命令行、仅通过 GUI 稳定完成目标站点下载时，视为第一阶段完成。

## 十七、风险

1. Cookie 读取在不同浏览器和系统上会有差异，尤其是 macOS。
2. 站点行为变化仍可能导致短期解析失效。
3. `yt-dlp` 与 `ffmpeg` 的打包和安装策略需要谨慎选择。
4. 后处理日志既要足够可排查，也不能让界面过度复杂。

## 十八、已确认决策

以下决策在本设计中已确定：

- 新建独立仓库
- 产品形态为桌面 GUI 应用
- 以个人使用和成功率优先
- 第一版优先支持 `x.com` 与 `pornhub.com`
- 同时提供“下载最佳”和“高级格式选择”
- 默认优先自动读取浏览器 Cookie
- 同时支持手动导入 Cookie，并提供获取指引
- 界面统一使用 React Spectrum

## 十九、成功标准

如果满足以下条件，则视为第一版设计目标达成：

- 能从 GUI 中粘贴并解析支持站点 URL
- 大多数情况下能一键下载最佳可用版本
- 在默认策略不满足时，用户可以手动选择高级格式
- Cookie 相关失败可以在应用内恢复
- 下载和合流进度可见
- 失败问题不需要打开开发者工具也能定位大致原因
